# 技术特色

## 四大核心特色

### 1. 文档驱动开发

- 以"地图/模板/检查点"为载体，确保过程可追溯、质量可度量
- check-in 作为实施前门禁，强制输出：问题、上下文、方案、边界、产物、标准、下一跳
- digest + update-map 形成治理闭环

### 2. 技能系统架构（五层）

| 层次 | 技能 | 职责 |
|------|------|------|
| 入口层 | origin, pipeline | 任务识别与分流 |
| 定义层 | pm, prd, req, check-in | 从模糊意图到清晰任务 |
| 交付层 | architect, qa, coder, audit | 设计、验证、实现、审计 |
| 治理层 | digest, update-map | 经验沉淀与状态更新 |
| 辅助层 | explore, init-docs, browser-verify, resolve-conflicts | 探索、初始化、验收 |

**执行骨架**：route → define(按需) → check-in → design(按需) → build → closeout

### 3. Web3 专用约束

- **审计评分**：满分 100，>=80 通过，60-79 软拒绝，<60 直接拒绝
- **轻审/重审**：FEAT 默认重审，PATCH 默认轻审
- **一票否决**：严重安全问题、关键不变量破坏、高风险边界缺失

### 4. Vibe Coding（自愈闭环）

- Coder 最多 10 轮自愈循环：实施 → 验证 → 根因分析 → 修复
- FEAT 中 QA 先执行 RED，Coder 职责是把 RED 变 GREEN
- 超过 10 轮输出 STUCK 报告，请求人工介入
- 发现范围扩大时回退 req/check-in/architect

## 解决的痛点

| 痛点 | 方案 |
|------|------|
| 流程僵化 | 任务类型分流 + 按需执行 |
| 文档与执行脱节 | check-in 强制对齐 |
| 变更失控 | 门禁 + 边界约束 |
| 质量评估主观 | audit 评分维度与阈值 |
