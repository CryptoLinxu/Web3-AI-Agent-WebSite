# Check-In 门禁技能

**职责**：实施前对齐点，确保问题边界、方案与完成标准清晰后才能进入实施环节。

## 输入/输出

- **输入**：来自 req 的任务卡（或 pm/prd 的范围定义）
- **输出**：结构化对齐模板（见下方）

## 适用场景

| 场景 | 是否强制 |
|------|----------|
| DELIVER-FEAT | 强制 |
| DELIVER-PATCH | 强制 |
| DELIVER-REFACTOR | 强制 |
| DEFINE（准备进入实施） | 强制 |
| DISCOVER / BOOTSTRAP / 纯 VERIFY/GOVERN | 不强制 |

## 输出模板

1. 本阶段要解决的问题
2. 本阶段必须掌握的上下文
3. 本阶段采用的方案
4. **本阶段不做什么**
5. 本阶段产物
6. **本阶段完成标准**
7. 进入下一阶段前要调用的 skill

## 硬规则

| 规则 | 说明 |
|------|------|
| 没有 check-in 不进入 architect/qa/coder | 硬门禁 |
| 必须明确"不做什么" | 否则视为未完成 |
| 必须明确完成标准 | 否则视为未完成 |
| 不代替 architect/qa/coder | 仅负责对齐，不负责实施 |

## 风险控制

| 风险 | 防止方式 |
|------|----------|
| 直接上手写代码 | 要求先明确问题、边界、方案 |
| PATCH 只修表象 | 强调"不做什么"与完成标准 |
| REFACTOR 不谈等价 | 要求明确等价性与风险边界 |
| FEAT 边界蔓延 | 通过"不做什么"收敛范围 |
