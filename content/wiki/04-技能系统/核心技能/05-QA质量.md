# QA 质量保证技能

**职责**：将完成标准转化为可执行的验证清单与测试策略，确保在实现前明确测试边界与通过条件。

## 输入/输出

- **输入**：check-in 完成标准、architect 架构说明、req 任务卡
- **输出**：测试清单、红灯结果或验证结果、回归检查点

## 两种工作模式

| 模式 | 适用场景 | 行为 |
|------|----------|------|
| RED 模式 | FEAT | 先写失败测试，证明当前未通过，RED 最多运行 2 次 |
| VERIFY 模式 | PATCH/REFACTOR | 轻量验证或回归验证，保障回归风险可控 |

## 红绿规则

| 规则 | 说明 |
|------|------|
| FEAT 先红后绿 | QA 阶段负责 RED，Coder 阶段负责 GREEN |
| RED 意外通过 | 需反思测试强度并修正 |
| 红灯与需求矛盾 | 停止并报告，不私自改需求 |
| 必须引用 check-in 完成标准 | 否则视为未完成 |
| PATCH 至少保留轻量回归检查 | 不可完全省略 |
| REFACTOR 优先保障回归验证 | 确保行为等价 |

## 核心规则

| 规则 | 说明 |
|------|------|
| 不直接写业务实现 | 只定义验证策略 |
| 不扩大需求范围 | 严格遵循 check-in 边界 |
| 衔接 | FEAT 进入 coder；PATCH/REFACTOR 进入 coder 或 audit |

## 测试金字塔

| 层级 | 面向 |
|------|------|
| 单元层 | 接口契约与关键算法 |
| 集成层 | 模块边界与消息流 |
| E2E 层 | 用户场景与验收标准 |
