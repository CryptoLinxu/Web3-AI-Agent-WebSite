# Audit 安全审计技能

**职责**：交付前最后一道风险关卡，不负责继续实现，只负责判断是否放行。

## 输入/输出

- **输入**：check-in 完成标准、代码结果、QA 结果、架构说明
- **输出**：审计结论（通过/软拒绝/拒绝）、总分、主要问题、风险建议

## 审计模式

| 模式 | 适用场景 |
|------|----------|
| 轻审 | 低风险 PATCH/低风险 REFACTOR |
| 重审 | FEAT、高风险 PATCH/REFACTOR、涉及权限/资金/安全的任务 |

## 评分与阈值

| 维度 | 权重 | | 阈值 | 结果 |
|------|------|-|------|------|
| 需求一致性 | 25 | | >=80 | 通过 -> digest |
| 结构/契约一致性 | 15 | | 60-79 | 软拒绝 -> 回退 coder |
| 安全与风险边界 | 20 | | <60 | 直接拒绝 -> 终止，人工介入 |
| 代码质量 | 15 | | | |
| 回归风险控制 | 10 | | | |
| 文档与状态收尾 | 10 | | | |
| 场景特定治理项 | 5 | | | |

## 一票否决项

- 严重安全问题
- 明显越过 check-in 的非目标
- 关键不变量被破坏
- 高风险场景缺少风险提示或失败降级

## 核心规则

| 规则 | 说明 |
|------|------|
| 不直接改代码 | 只判断是否放行 |
| 不重新定义需求 | 依据 check-in 标准核验 |
| Web3 场景额外关注 | 签名权限、资金路径、数据可信度、Gas 处理 |
