# 工具抽象层设计

## 设计理念

将 Web3 数据查询抽象为标准化工具,统一输入/输出契约、错误处理与降级策略,保证 Agent 在不同数据源间平滑切换。

## 统一接口规范

每个工具遵循相同的调用生命周期: **参数校验 -> 执行查询 -> 结果格式化 -> 错误处理**

```typescript
// 标准工具结果接口
interface ToolResult<T> {
  success: boolean;
  data?: T;
  error?: string;
  timestamp: string;   // ISO 8601
  source: string;      // 数据来源标识
}
```

工具属性: 工具名称、输入参数、输出结构、错误码、降级策略、数据来源标识。

## 工具注册与路由

工具通过技能系统注册,由 origin 入口路由分发:

| 任务类型 | 管道 | 必经环节 |
|---------|------|---------|
| FEAT | 全链路 | prd->req->check-in->architect->qa->coder->audit->digest->update-map |
| PATCH | 快速链路 | req->check-in->coder->qa->digest->update-map |
| REFACTOR | 设计优先 | req->check-in->architect->qa->coder->audit->digest->update-map |

## 参数校验规则

| 场景 | 处理 |
|------|------|
| 缺少必填参数 | 返回错误 + 参数说明 |
| 参数格式无效 | 返回错误 + 合法格式示例 |
| 不支持的值 | 返回错误 + 支持列表 |

## 错误处理策略

| 错误类型 | 处理方式 |
|---------|---------|
| 参数校验失败 | 返回 "参数无效" + 建议 |
| 外部 API 超时/异常 | 自动切换备用数据源; 全失败则返回明确错误 |
| 网络不可用 | 返回 "网络不可用" + 重试建议 |

核心原则: 失败时**绝不伪造数据**, 明确标注来源或 "数据获取失败"。

## 质量保障机制

- **QA 红绿灯**: FEAT 先 RED (证明未通过), 再由 coder 变为 GREEN
- **Coder 自愈**: 最多 10 轮自愈循环; 超限输出 STUCK 报告
- **Audit 评分**: 总分 100; >=80 通过, 60-79 回退修复, <60 拒绝

## 扩展新工具流程

1. 定义工具名称、输入参数、输出结构 (遵循 ToolResult 接口)
2. 配置数据源 (支持多源并行与故障转移)
3. 实现参数校验与结果格式化
4. 编写测试用例与异常路径验证
5. 在技能系统中注册,确保 Agent 可调用
