# 文档驱动开发

> 本项目的核心理念：以文档为先，再进入编码。

---

## 🎯 什么是文档驱动开发？

**文档驱动开发**（Document-Driven Development）是一种开发方法论：

- 在进入编码前，先完成需求、架构、测试与风险评估
- 通过标准化流程、阶段性门禁与约束，降低试错成本、提升交付质量
- 以文档作为团队沟通和知识沉淀的主要载体

### 传统开发 vs 文档驱动开发

| 维度 | 传统开发 | 文档驱动开发 |
|------|---------|-------------|
| **起点** | 直接编码 | 文档定义 |
| **需求** | 口头沟通或简单注释 | PRD + 需求拆解 |
| **架构** | 边做边想 | 架构设计文档 |
| **测试** | 事后补充 | 先写验证清单 |
| **风险** | 遇到问题再解决 | 提前评估与规避 |
| **知识沉淀** | 依赖个人记忆 | 文档化复盘 |

---

## 📋 文档驱动的完整流程

```
想法/需求
    ↓
PM（产品意图整理）→ PRD（产品需求文档）→ REQ（需求拆解）
    ↓
Check-In（实施前对齐）
    ↓
Architect（架构设计）→ QA（验证清单）
    ↓
Coder（实现编码）→ Audit（安全审计）
    ↓
Digest（复盘沉淀）→ Update-Map（状态更新）
```

### 每个阶段的文档产物

| 阶段 | 技能 | 产物文档 | 目的 |
|------|------|----------|------|
| **需求定义** | PM | `pm.md` | 整理产品意图、价值主张 |
| | PRD | `prd.md` | 定义正式范围、非目标、验收标准 |
| | REQ | `req.md` | 拆成最小可执行任务卡 |
| **实施前对齐** | Check-In | `check-in.md` | 确认问题、上下文、方案、完成标准 |
| **架构设计** | Architect | `design.md` | 结构设计、接口定义、模块边界 |
| **质量保证** | QA | `qa.md` | 验证清单、测试用例 |
| **实现编码** | Coder | 代码实现 | 按照设计文档编码 |
| **安全审计** | Audit | `audit.md` | 风险审计、评分 |
| **复盘沉淀** | Digest | `digest.md` | 完成项、问题、经验、后续建议 |
| **状态更新** | Update-Map | 项目状态 | 更新索引、下一步入口 |

## 🚦 门禁机制

### 为什么需要门禁？

门禁（Gate）确保每个阶段的质量，防止问题传递到下一阶段。

### Check-In 门禁

**实施前对齐点**，只有实施型任务强制进入：

- DELIVER-FEAT（新功能开发）
- DELIVER-PATCH（Bug 修复）
- DELIVER-REFACTOR（重构）
- DEFINE 中准备进入实施的任务

**7 项输出结构**：
1. **问题**：要解决什么问题？
2. **上下文**：相关背景信息？
3. **方案**：如何解决？
4. **不做项**：明确不做什么？
5. **产物**：交付什么文档/代码？
6. **完成标准**：如何判断完成？
7. **下一 skill**：下一步进入哪个技能？

### QA 红绿灯规则

- **FEAT**：默认先由 QA 执行 RED（先证明"当前未通过"）
- **PATCH/REFACTOR**：保留验证或回归检查

### Audit 评分规则

总分 100 分：
- `>= 80`：通过
- `60-79`：软拒绝，回退修复
- `< 60`：直接拒绝，方案废弃
- 严重安全问题可一票否决

## 📁 文档组织结构

### 技能文档

每个技能独立的 SKILL.md 文件，定义其职责、输入输出和执行规则：

```
skills/x-ray/
├── SKILL-SYSTEM-DESIGN-V3.md  # 总体设计
├── MAP-V3.md                  # 技能地图
├── COMMANDS.md                # 命令约定
├── origin/
│   └── SKILL.md               # Origin 技能定义
├── pm/
│   └── SKILL.md               # PM 技能定义
├── prd/
│   └── SKILL.md               # PRD 技能定义
└── ...
```

### 产物文档

任务执行过程中生成的文档，记录决策和上下文：

```
docs/
├── prd/
│   └── prd.md                 # 产品需求文档
├── req/
│   └── req.md                 # 需求拆解卡片
├── architect/
│   └── design.md              # 架构设计文档
├── changelog/
│   └── 2026-05-08-feat-xxx.md # 变更记录
├── digest/
│   └── digest.md              # 复盘记录
└── checklist/
    └── PROJECT-CHECKLIST.md   # 项目清单
```

## 🎓 文档驱动的优势

### 1. 降低试错成本

- 在编码前发现需求不清、方案不可行等问题
- 避免"做了半天发现方向错了"

### 2. 提升交付质量

- 架构设计确保结构合理
- QA 验证清单确保功能完整
- Audit 审计确保安全性

### 3. 知识沉淀

- 每个任务的决策和上下文都记录在文档中
- 新人可以通过文档快速了解项目历史
- AI 可以通过文档了解项目现状

### 4. 团队协作

- 文档作为沟通媒介，减少口头沟通的信息丢失
- 明确的完成标准，减少理解偏差

### 5. 可追溯性

- 从需求到实现，每个环节都有文档记录
- 出现问题时可以快速追溯原因

## 🛠️ 实践建议

### 对于个人开发者

1. **从小处开始**：不要一开始就写完整的 PRD，先从简单的需求描述开始
2. **保持文档更新**：每次任务完成后，更新相关文档
3. **善用 AI**：让 AI 帮你生成文档初稿，你负责审核和修改

### 对于团队

1. **制定文档规范**：统一文档格式、命名规范、存放位置
2. **强制执行门禁**：不经过 Check-In 的任务不允许进入编码
3. **定期回顾**：定期检查文档质量，持续改进

### 对于 AI Agent

1. **文档作为上下文**：AI 通过读取文档了解项目现状
2. **文档作为产物**：AI 生成的文档记录决策和实现细节
3. **文档作为验证**：AI 通过文档验证任务是否完成
