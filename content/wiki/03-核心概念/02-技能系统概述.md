# 技能系统概述

> Web3 AI Agent 的核心是一套可路由、可裁剪、可回退的技能操作系统。

---

## 🎯 技能系统设计理念

### 文档驱动开发

本项目采用**文档驱动开发**理念：
- 在进入编码前，先完成需求、架构、测试与风险评估
- 通过标准化流程、阶段性门禁与约束，降低试错成本、提升交付质量

### 分层路由架构

技能系统采用五层架构设计：

```
入口层 → 定义层 → 交付层 → 治理层 → 辅助层
```

| 层级 | 职责 | 包含技能 |
|------|------|----------|
| **入口层** | 任务识别与路由 | Origin, Pipeline |
| **定义层** | 需求澄清与对齐 | PM, PRD, REQ, Check-In |
| **交付层** | 架构、实现与质量保证 | Architect, QA, Coder, Audit |
| **治理层** | 知识沉淀与状态维护 | Digest, Update-Map |
| **辅助层** | 探索、初始化与验证 | Explore, Init-Docs, Browser-Verify, Resolve-Conflicts |

---

## 📋 任务分类与路由

系统支持 7 类任务类型：

| 任务类型 | 适用场景 | 推荐链路 |
|---------|---------|----------|
| **DISCOVER** | 熟悉项目、查询模块、定位代码 | origin → explore |
| **BOOTSTRAP** | 新项目初始化、建立文档体系 | origin → init-docs → update-map |
| **DEFINE** | 目标模糊、需求未收敛 | origin → pm/prd/req → check-in |
| **DELIVER-FEAT** | 新功能、新模块 | origin → pipeline(FEAT) → 完整交付链 |
| **DELIVER-PATCH** | Bug 修复、回归修复 | origin → pipeline(PATCH) → 快速交付链 |
| **DELIVER-REFACTOR** | 重构、模块拆分 | origin → pipeline(REFACTOR) → 设计优先链 |
| **VERIFY/GOVERN** | 验收、审计、文档治理 | origin → qa/audit/browser-verify |

---

## 🔀 交付 Pipeline

### FEAT Pipeline（完整流程）

用于新功能开发，包含所有质量门禁：

```
origin → pipeline(FEAT) → pm(按需) → prd → req → check-in → architect → qa → coder → audit → digest → changelog → project-checklist → update-map
```

### PATCH Pipeline（快速流程）

用于 Bug 修复，跳过部分环节：

```
origin → pipeline(PATCH) → req → check-in → coder → qa → digest → changelog → project-checklist → update-map
```

按需插入：architect（涉及结构变化）、audit（涉及安全/资金）

### REFACTOR Pipeline（设计优先）

用于重构任务，强调设计先行：

```
origin → pipeline(REFACTOR) → req → check-in → architect → qa → coder → audit(轻/重) → digest → changelog → project-checklist → update-map
```

## 🚦 质量门禁机制

### Check-In 门禁

实施前对齐点，确保：
- 问题定义清晰
- 上下文完整
- 方案可行
- 完成标准明确

**强制进入**：所有实施型任务（FEAT/PATCH/REFACTOR）必须经过 check-in

### QA 红绿灯规则

- **FEAT**：默认先由 QA 执行 RED（先证明"当前未通过"）
- **PATCH/REFACTOR**：保留验证或回归检查

### Audit 评分规则

总分 100 分：
- `>= 80`：通过
- `60-79`：软拒绝，回退修复
- `< 60`：直接拒绝，方案废弃
- 严重安全问题可一票否决

### Coder 自愈规则

- Coder 负责将 QA 的 RED 变为 GREEN
- **最多 10 轮自愈循环**，超过必须终止并人工介入

## 🛠️ 核心技能详解

### Origin（统一入口）

**职责**：任务识别与路由

**规则**：
- 任何任务必须从 origin 进入
- 不允许跳过 origin 直接调用子 skill
- 负责任务分类，再路由到对应链路

### Pipeline（交付分流）

**职责**：在 FEAT/PATCH/REFACTOR 之间选择执行深度

**执行深度**：
- **FEAT**：完整流程，包含所有质量门禁
- **PATCH**：快速流程，跳过部分环节
- **REFACTOR**：设计优先，强调架构设计

### Check-In（实施前对齐）

**职责**：确保实施前所有边界清晰

**7 项输出结构**：
1. 问题定义
2. 上下文
3. 方案
4. 不做项
5. 产物
6. 完成标准
7. 下一 skill

### Architect（架构设计）

**职责**：结构设计、接口定义、模块边界

**触发条件**：涉及结构变化、新模块、重构

### QA（质量保证）

**职责**：将完成标准转成验证清单

**执行方式**：
- FEAT：红灯优先（先证明未通过）
- PATCH/REFACTOR：轻量验证或回归验证

### Coder（实现编码）

**职责**：在边界清晰的前提下实施代码

**自愈循环**：最多 10 轮，将 QA 的 RED 变为 GREEN

### Audit（安全审计）

**职责**：对实现结果做风险审计和评分

**审计类型**：
- **轻审**：常规代码审查
- **重审**：涉及安全、资金的深度审计

## 🔄 辅助技能

### Explore（探索发现）

**职责**：只读探索，快速理解项目、定位模块、查询现状

**特点**：不进入交付链，纯信息查询

### Init-Docs（文档初始化）

**职责**：新项目初始化文档体系、首次建立地图

**触发场景**：新项目、旧文档迁移

### Digest（复盘沉淀）

**职责**：任务完成后做阶段沉淀

**输出内容**：
- 完成项
- 问题
- 经验
- 后续建议

### Update-Map（地图更新）

**职责**：更新项目状态、索引、下一步入口

**触发场景**：pipeline 执行完毕后、changelog 记录后

### Browser-Verify（浏览器验证）

**职责**：用浏览器层做可视化和交互验收

**适用场景**：前端、交互和页面回归验证

### Resolve-Conflicts（文档冲突解决）

**职责**：专门处理文档冲突

**优先级**：优先清理 docs 类冲突，避免与代码修复混在一起

## 📊 技能协作机制

### 技能间通信

技能之间通过**文档产物**进行通信：

```
PM → PRD → REQ → Check-In → Architect → QA → Coder → Audit → Digest
  ↓      ↓      ↓        ↓          ↓        ↓       ↓        ↓       ↓
prd.md  req.md  check-in.md  design.md  qa.md   code    audit.md digest.md
```

### 阶段模型

每个技能执行后生成标准化产物，下一个技能基于产物继续执行。

**关键产物**：
- `prd.md`：产品需求文档
- `req.md`：需求拆解卡片
- `check-in.md`：实施前对齐记录
- `design.md`：架构设计文档
- `qa.md`：验证清单
- `audit.md`：审计报告
- `digest.md`：复盘记录

## 🌐 Web3 专属约束

作为 Web3 AI Agent，技能系统内置以下约束：

### 数据来源约束

- Agent 核心为 "LLM + 工具 + 循环 + 记忆"
- 强调数据来源与风险边界
- 不可伪造链上实时数据

### 安全约束

- 涉及资金操作必须经过 Audit
- 钱包相关代码需要额外验证
- 链上交易需要二次确认

### 用户体验约束

- 工具调用结果需要结构化展示
- 错误信息需要用户友好
- 加载状态需要明确反馈
